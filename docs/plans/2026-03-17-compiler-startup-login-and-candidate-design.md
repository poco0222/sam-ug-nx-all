# 编制人启动链路与接口日志重构设计

## 背景

当前仓库中“当前编制人”的启动链路分散在 Qt 宿主与前端两侧：

- Qt 宿主会在本地文件、宿主上下文都缺失时回退到环境变量用户名
- Qt 宿主会主动调用 `getBianChengUserInfo` 构建候选编制人列表
- 前端收到 context 后会立刻应用上下文，并在此过程中触发一次 `/loginByFree`

这套流程有三个问题：

1. 默认用户名会受宿主操作系统账号影响，不稳定
2. 候选用户接口请求藏在宿主中，前端难以精确控制时序和日志
3. 免登录与候选用户接口缺少落盘日志，现场排查成本高

本次目标是统一启动链路，使默认用户名稳定为 `PopoY`，将候选用户接口收归前端控制，并把启动阶段两个关键接口的完整日志写入 `D:/UGerkaiInfo/userInfo/APIlog`。

## 设计目标

- 默认用户名稳定，且不再依赖宿主环境变量
- 宿主不再主动获取候选用户列表
- 前端在拿到已解析用户名后，立即并行执行：
  - `/loginByFree`
  - `getBianChengUserInfo`
- 免登录优先级提高：只要拿到已有用户名，就立刻发起免登录
- `APIlog` 每次宿主启动时先清空，仅保留本次运行日志
- 日志详细记录两个接口的开始、成功、失败、耗时、关键入参和返回内容

## 方案选择

### 方案 A：宿主仅负责“当前用户名 + 日志落盘”，前端负责“免登录 + 候选用户接口”

这是本次采用的方案。

核心思路：

- Qt 宿主继续负责解析“当前用户名”，但不再请求候选用户接口
- Qt 宿主在启动时清空 `APIlog`，并提供追加写日志的桥接方法
- 前端收到 context 后，先解析当前用户名，再并行触发免登录与候选用户接口
- 前端以接口返回的候选用户列表作为下拉主数据源

优点：

- 完全符合“不要去宿主里获取候选用户”的要求
- 候选用户获取与免登录的时序可在前端显式管理
- 接口日志既能包含前端请求细节，又能稳定落到本地文件

代价：

- 需要同时调整 Qt 宿主桥接和前端启动链路

### 方案 B：全部改由宿主统一请求并记录日志

优点是实现简单，但不符合“候选用户不要走宿主”的要求，因此不采用。

### 方案 C：前端请求接口，但日志仍只打在浏览器控制台

优点是改动少，但现场排查时难以留痕，也不满足日志落盘要求，因此不采用。

## 详细设计

### 1. 当前用户名解析优先级

Qt 宿主中的当前用户名解析调整为：

1. 本地 `userName.txt`
2. 宿主 `upload-data-file` / 上下文中已有用户名
3. 固定默认值 `PopoY`

其中：

- 不再使用系统环境变量 `USERNAME` / `USER` / `LOGNAME` 作为默认用户名来源
- `Administrator -> admin` 的环境变量规范化逻辑也随之退场，因为该分支不再承担默认兜底职责

### 2. 候选用户来源

Qt 宿主不再调用 `getBianChengUserInfo`。

宿主只负责把当前用户名下发到 context 中的 `createBy` / `operator` 等字段，供前端识别“当前编制人”。

前端新增候选用户状态，启动后以接口返回结果为主：

- 成功：使用接口返回的候选用户列表
- 失败：至少保留当前用户名这一项，确保页面可操作

### 3. 启动时序

前端接收到宿主 context 后，启动顺序调整为：

1. 从 context 中提取当前用户名
2. 如果取不到用户名，则回退为 `PopoY`
3. 立刻并行启动两个请求：
   - `/loginByFree`
   - `getBianChengUserInfo`
4. 免登录成功后立即写入 `sam_token`
5. 再将 context 应用到 store，维持现有页面数据加载依赖 token 的行为
6. 候选用户接口返回后单独更新编制人下拉候选项

这样可以保证：

- 页面主业务数据仍然在 token 就绪后再进入原有加载链路
- 候选用户接口不会阻塞页面首包处理

### 4. APIlog 设计

日志文件固定为：

- `D:/UGerkaiInfo/userInfo/APIlog`

宿主启动时：

- 若目录不存在则创建
- 先清空 `APIlog`

日志写入方式：

- 前端将接口日志文本通过桥接方法传给宿主
- 宿主以追加方式写入 `APIlog`

日志内容覆盖两个接口：

- `/loginByFree`
- `getBianChengUserInfo`

每个接口记录三类事件：

- `START`
- `SUCCESS`
- `FAIL`

每条日志至少包含：

- 时间戳
- 接口名称
- URL
- HTTP 方法
- 当前用户名
- 请求参数
- 耗时
- 响应状态码
- 响应体或错误信息

### 5. 失败退化

#### 免登录失败

- 记录失败日志
- 页面提示免登录失败
- 不继续自动加载依赖 token 的业务数据

#### 候选用户接口失败

- 记录失败日志
- 候选列表退化为“当前用户名单项”
- 页面其它功能不因此阻塞

#### 日志写文件失败

- 宿主桥接返回失败结果
- 前端仅做控制台告警，不阻断业务流程

## 验证策略

### 用户名优先级

- 无本地文件、无宿主用户名时，当前用户名应为 `PopoY`
- 有本地文件时，应优先使用文件中的用户名

### 启动时序

- 收到 context 后，先发起免登录，再进入依赖 token 的页面数据加载
- 候选用户接口与免登录并行执行

### 候选用户

- 宿主不再调用候选用户接口
- 前端接口成功时，下拉显示接口返回用户
- 接口失败时，下拉至少保留当前用户名

### 日志

- 每次宿主启动后 `APIlog` 被清空
- 免登录与候选用户接口都有完整的开始/成功/失败日志
- 日志写入失败不影响主业务流程
